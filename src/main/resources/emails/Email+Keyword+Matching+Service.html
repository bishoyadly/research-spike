<p><div class="toc-macro client-side-toc-macro conf-macro output-block" data-headerelements="H1,H2,H3,H4,H5,H6,H7" data-hasbody="false" data-macro-name="toc"> </div></p><h2 id="EmailKeywordMatchingService-Overview"><strong>Overview</strong></h2><p>keyword matching is a major requirement for email routing execution, it provide routing services with attributes extracted from email based on predefined keywords and rules which identify nature of incoming email interaction.</p><p>the main functionality of keyword matching service is to evaluate set of rules, each rule is based on a group of predefined keyword, and if rule is valid, then values assigned to system attributes will be return as result to be used to rout email to correct agent or context.</p><p><br/></p><p><span class="confluence-embedded-file-wrapper confluence-embedded-manual-size"><img class="confluence-embedded-image" height="250" src="/download/attachments/199506789/image2020-7-16_20-55-6.png?version=1&amp;modificationDate=1594926578354&amp;api=v2" data-image-src="/download/attachments/199506789/image2020-7-16_20-55-6.png?version=1&amp;modificationDate=1594926578354&amp;api=v2" data-unresolved-comment-count="0" data-linked-resource-id="199506762" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="image2020-7-16_20-55-6.png" data-base-url="https://confluence.forge.avaya.com" data-linked-resource-content-type="image/png" data-linked-resource-container-id="199506789" data-linked-resource-container-version="16"></span></p><p><br/></p><p>one basic assumption here is that, based on order of execution for rules and assuming that service will return list of attributes with a final value for each attribute. values of attributes can override each other based on rules execution order.</p><p><span class="confluence-embedded-file-wrapper confluence-embedded-manual-size"><img class="confluence-embedded-image" height="148" src="/download/attachments/199506789/image2020-7-16_21-18-41.png?version=1&amp;modificationDate=1594927121598&amp;api=v2" data-image-src="/download/attachments/199506789/image2020-7-16_21-18-41.png?version=1&amp;modificationDate=1594927121598&amp;api=v2" data-unresolved-comment-count="0" data-linked-resource-id="199506802" data-linked-resource-version="1" data-linked-resource-type="attachment" data-linked-resource-default-alias="image2020-7-16_21-18-41.png" data-base-url="https://confluence.forge.avaya.com" data-linked-resource-content-type="image/png" data-linked-resource-container-id="199506789" data-linked-resource-container-version="16"></span></p><p>below diagram describe service's basic entities and relation ship between each of them as below : </p><ul style="list-style-type: square;"><li><strong>Group</strong> will contain one or more Keyword to be evaluated in email message..</li><li><strong>Groups</strong> can be assigned to different <strong>Rules</strong>. and a <strong>Rule</strong> can have one or more <strong>Group</strong> assigned.</li><li><strong>Rule</strong> will define set of <strong>Attributes</strong>' values, which will return to caller if <strong>Keywords</strong> define in <strong>Rule</strong>'s <strong>Groups</strong> exits in email.</li></ul><h2 id="EmailKeywordMatchingService-HighlevelRequirements"><strong>High level Requirements</strong></h2><p>below are high level requirements for keyword matching service :</p><ul><li><p>Tenant Admin can configure lookup for specific keywords in email subject and body and based on that set the value of routing attributes</p></li><li><p>Keyword group is created grouping one or more keywords to be checked in incoming emails</p></li><li><p>Keyword rules are created to reflect a meaning from looking up of keyword groups exist or doesnt exist in an incoming email</p></li><li><p>Keyword rule is mapped to routing attributes</p></li><li><p>Multiple rules can be configured for an account to address multiple attributes</p></li><li><p>Rules will be executed in order. Final attributes are the union of all set attributes. Attributes values can be overridden according to execution order</p></li></ul><p>based on above requirements, Tenant's Admin will be responsible to define all aspects related to keyword matching service as below:<span class="confluence-embedded-file-wrapper conf-macro output-inline" style="" data-hasbody="true" data-macro-name="plantuml"><img class="confluence-embedded-image" src="/download/temp/plantuml2829460924122655025.png?contentType=image/png" style="" /></span><br/></p><p>as per above diagram Tenant's admin is capable to do the following :</p><ul style="list-style-type: square;"><li><em>Manage Keywords' Groups</em> : admin can add, update, and delete keywords groups and keywords within. however admin should receive a warning when trying to delete a keyword group assigned to rules.</li><li><em>Manage Keywords' Rules</em> : admin is capable to define, update, and delete keywords' rules, also admin will be able to set and update attribute values related to rule which consider final result for rule evaluation. </li></ul><p><br/></p><p>now regarding to the main functionality of keyword matching service, below diagram describe activities involved in handling a matching request:</p><div class="table-wrap"><table class="relative-table wrapped confluenceTable" style="width: 100.0%;"><colgroup><col style="width: 71.1626%;"/><col style="width: 28.7786%;"/></colgroup><tbody><tr><td class="confluenceTd"><ul><li>when a matching request sent to keyword matching service , it will include email body , subject , or both.<br/><br/></li><li>the most first validation will be to check if any rules have been defined by tenant admin for email messages' content, if none found service should return an error or an empty list of attributes based on integration scenario expected by ED team.<br/><br/></li><li>if tenant have any rules, first service will check email subject. Subject is always a plain text and can easily be Tokenized. <br/>Tokenized means we will analyze all keywords appeared in subject in a list of tokens, this list will help in validating rules' keywords. so instead of searching the whole text for each rule's keyword, service will lookup Tokens for rules keywords and if any exists rule will be marked as valid.<br/><br/><br/></li><li>on the other hand , if email content passes to matching service include body, e need first to extract text for email body considering that email body can be in a HTML or rich text content.<br/><br/></li><li>once email body's text is extracted, it will be Tokenized on the same way as Subject did, and extract words exist in body text.<br/>tokens will be saved in any Sorted data structure to be able to apply fast searching algorithm for better performance.<br/><br/></li><li>finally service will retrieve all matching rules related to tenant, and validate each rule words against a final list of tokens extracted form subject and/or body.<br/><br/></li><li>for any rule related one or more keyword group, each groups' keywords will validated against tokens, if any keyword exists in token list group will be marked as true for logical evaluation with other groups results.<br/><br/></li><li>one rule can have more than one keyword group, with logical operators to define evaluation logic of keywords groups. initially service will be implemented considering logical operator is OR &quot;if any keyword group exists in email content rule will be valid&quot;, and later in implementation stages operators will be saved and used STACK or Recursion to evaluate final keyword groups results.</li></ul></td><td class="confluenceTd"><div class="content-wrapper"><p><span class="confluence-embedded-file-wrapper conf-macro output-inline" style="" data-hasbody="true" data-macro-name="plantuml"><img class="confluence-embedded-image" src="/download/temp/plantuml1598949897562943295.png?contentType=image/png" style="width:200px" /></span></p></div></td></tr></tbody></table></div><div class="table-wrap"><table class="relative-table
